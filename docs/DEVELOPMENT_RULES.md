# Свод правил разработки Mudaroba

Правила учитывают специфику проекта: Next.js 14 (pages router, i18n ru/en, defaultLocale=ru) + Django/DRF + Celery + Postgres + Redis + R2/Cloudflare, мультидоменный каталог (18 типов товаров + услуги), деплой docker-compose за nginx и Cloudflare.

---

## 1. SEO — железные правила

Это e-commerce: страницы каталога — главный источник трафика. Любая правка страниц обязана соблюдать:

1. **Роутинг локалей:** `ru` — без префикса (`/product/x`), `en` — с префиксом (`/en/product/x`). URL вида `/ru/...` Next отдаёт как 200-дубликат (склеивается 301-ом на уровне nginx). Никогда не генерировать ссылки/редиректы/canonical с `/ru`.
2. **Canonical:** всегда self-referencing с учётом локали:
   ```ts
   const localePrefix = router.locale === router.defaultLocale ? '' : `/${router.locale}`
   const canonicalUrl = `${siteUrl}${localePrefix}${path}`
   ```
   Захардкоженный canonical без префикса = EN-версия выпадает из индекса.
3. **hreflang:** на каждой индексируемой странице ровно три тега: `ru` (без префикса), `en` (`/en/...`), `x-default` (= ru). Один источник — страница; в `_document.tsx` hreflang не дублировать.
4. **Редиректы в getServerSideProps:** `destination` строить через `ctx.locale !== ctx.defaultLocale ? '/'+ctx.locale : ''`. Постоянный редирект (`permanent: true`) — только когда URL действительно навсегда сменился.
5. **Ошибки бэкенда ≠ 404.** `notFound: true` возвращать только если API явно ответил 404. Сетевые ошибки и 5xx — пробрасывать (бот получит 500 и вернётся, страница останется в индексе).
6. **noindex** — на cart, checkout, profile, auth, favorites, search, страницах пагинации `page>1`. На noindex-страницах canonical/hreflang не нужны.
7. **Sitemap:** единственный источник — `frontend/src/pages/sitemap.xml.tsx`. Новая сущность с публичной страницей → сразу добавить в sitemap (с hreflang-альтернативами) и проверить canonical.
8. **Метаданные двуязычны.** Title/description берутся из переводов API (`translations[locale]`), фолбэк — шаблонные строки через `t(...)`. Не хардкодить русский текст в мету.
9. **JSON-LD:** карточка товара — `Product` (+ `Offer`, `aggregateRating` при наличии отзывов); категории — `BreadcrumbList`; главная — `Organization` + `WebSite`. Валидировать через Rich Results Test после изменения.
10. **Перед мержем любой правки страниц** прогнать чек: `curl -s https://site/<page> | grep -E 'canonical|hreflang|robots'` для обеих локалей.

## 2. Производительность

1. **SSR — только то, что нужно для первого экрана и меты.** Остальное (рекомендации, отзывы, похожие товары) — клиентская подгрузка.
2. **Кэшируй HTML:** каждая публичная gSSP-страница ставит `Cache-Control: public, s-maxage=..., stale-while-revalidate=...`. Персонализация (валюта, корзина) — на клиенте, не в SSR-HTML.
3. **Параллель, не серия:** несколько независимых запросов к API в gSSP — только `Promise.all/allSettled` с таймаутами. Циклы по страницам пагинации в SSR запрещены.
4. **Запросы к БД:** новый list/detail-эндпоинт обязан иметь `select_related`/`prefetch_related` и `only()` по факту использования сериализатором. Проверка — `django-debug-toolbar`/`assertNumQueries` в тесте.
5. **Медиа:** изображения — через `cdn.mudaroba.com` или proxy-media с `max_width`; в `<Image>` указывать `sizes`; `priority` — только для LCP-элемента. Видео никогда не стримить через Django, если есть прямой CDN-URL.
6. **Кэш Redis:** ключи версионировать (`v2:...`), не вайпить кэш целиком при деплое. TTL обязателен.
7. **Длинные операции — в Celery**, не в request-цикле (генерация, парсинг, рассылки, конвертация).

## 3. Бэкенд (Django/DRF)

1. Каждый публичный эндпоинт: пагинация (`StandardPagination`), фильтры через query-параметры, `max_page_size` ≤ 1000.
2. Локализация ответа — по `Accept-Language`; новые текстовые поля моделей сразу получают перевод (механизм `translations`).
3. Списки типов товаров (`BASE_PRODUCT_TYPES`, `TYPES_NEEDING_PATH`) — единый источник правды на бэкенде; фронт не копирует, а потребляет (до рефакторинга: менять синхронно в `product_resolve.py`, `lib/product.ts`, `sitemap.xml.tsx` — все три, и обновлять тесты).
4. Слаги валидируются на входе (парсеры, админка, AI): нижний регистр, дефисы, без дублирующихся половин, уникальность по всем доменным моделям.
5. Миграции — только через `makemigrations` локально и в коммит; на проде только `migrate`.
6. Никаких секретов в коде и дефолтах настроек; прод падает при отсутствующем env, а не работает на заглушке.

## 4. Фронтенд (Next.js)

1. Страница > ~400 строк — выделяй компоненты/хуки. В `pages/` — только роутинг, мета и gSSP; логика — в `components/`, `lib/`, `hooks/`.
2. Мета-теги — через единый компонент (после починки `SEO.tsx`); ручной `<Head>` с canonical в новых страницах — запрещён.
3. Переводы: каждая новая строка UI — сразу в `public/locales/ru/common.json` и `en/common.json`; `t('key', 'фолбэк')` обязателен.
4. Клиентские запросы — через `lib/api.ts` (`getClientApiBase`), SSR — `getInternalApiUrl`. Прямые URL бэкенда в компонентах запрещены.
5. Деньги: форматирование только через `lib/price.ts`. Никаких локальных `toFixed`/`toLocaleString`.

## 5. E-commerce правила

1. **Цена и наличие правдивы.** JSON-LD `Offer` и видимая цена обязаны совпадать; нет цены — `price on request`, без нулей.
2. **Деньги — Decimal на бэкенде**, никогда float. Округление вверх (`Math.ceil`) — только на отображении, не в расчётах заказа.
3. **Снятый с продажи товар не 404:** страница остаётся (200) с пометкой «нет в наличии» и ссылками на аналоги; 410 — только при окончательном удалении; редирект 301 — при замене на преемника.
4. **Смена slug = 301-редирект** со старого URL (механизм `canonical_path` resolve уже есть — поддерживать).
5. **Checkout-цепочка неприкосновенна:** изменения cart/checkout/payments требуют ручной проверки на стейдже (карта + крипто) и юнит-тестов на пересчёт суммы.
6. **Аналитика событий e-commerce** (view_item, add_to_cart, purchase) не ломается молча: правки `gtm.ts`/`gtag.ts`/`ym.ts` проверять в Preview-режиме GTM.
7. **Фиды (YML и т.п.)** меняются вместе с моделью товара: добавил поле/тип — проверь, что фиды и sitemap его учитывают.

## 6. Git

1. **main — защищённая ветка.** В main — только через PR (или merge ветки после ревью), никогда напрямую при работе над фичей. Hotfix-исключение допустимо, но фиксируется тегом.
2. **Ветки:** `feature/<краткое-имя>`, `fix/<краткое-имя>`, `seo/<...>`, `perf/<...>`. Одна ветка — одна задача.
3. **Коммиты:** на русском, формат «Префикс: что сделано» — как уже принято (`Сигнал:`, `Перформанс:`, `SEO:`, `Починить:`). Один логический шаг = один коммит. Без Co-Authored-By агентов.
4. **CI обязателен** (когда появится — см. роадмап P1.7): lint + tsc + тесты + build. Красный CI = нет мержа.
5. **Теги релизов:** каждый прод-деплой — тег `vYYYY.MM.N`. Откат = деплой предыдущего тега.
6. **Не коммитить:** `.env*` (кроме `*.example`), дампы, медиа, `node_modules`, сгенерированные файлы. Новые MD-доки — в `docs/`, не в корень.
7. **PR-описание:** что изменилось, как проверено (команды/скрины), затронуты ли SEO-мета/фиды/checkout — отдельной строкой.

## 7. Тестирование и деплой

1. Багфикс начинается с теста, воспроизводящего баг.
2. Минимум для merge: `pytest` бэкенда зелёный, `npm run build` фронта проходит, ручная проверка изменённой страницы в обеих локалях.
3. SEO-правки дополнительно: проверка canonical/hreflang curl-ом на обеих локалях + одна карточка каждой затронутой категории.
4. Деплой не в пятницу вечером; после деплоя — смоук: главная, категория, карточка, поиск, корзина, /sitemap.xml.
5. После деплоя следить за Search Console (Coverage) минимум неделю при SEO-изменениях.
